# Secure & Scalable Task Manager API & Dashboard

An enterprise-grade, monorepo-structured task management application. The project features a robust **Express + TypeScript + Prisma (SQLite)** REST API with JWT security and role-based access control, integrated with a highly responsive, modern, dark-themed **React + Vite + Vanilla CSS** frontend dashboard.

## Architecture & Tech Stack

### Backend (REST API)
- **Runtime & Framework:** Node.js + Express with TypeScript.
- **Database ORM:** **Prisma ORM** with **SQLite**. Prisma acts as a database-agnostic interface, allowing developers to switch from SQLite to **PostgreSQL**, **MySQL**, or **MongoDB** by changing a single line of config in `schema.prisma`.
- **Security:**
  - JWT authorization tokens signed with `HS256`.
  - Password hashing via `bcryptjs` with 10 salt rounds.
  - Express middleware checks for validation limits (`express-rate-limit`) on authentication paths.
  - HTTP headers fortification using `helmet`.
  - Safe Cross-Origin Resource Sharing (`cors`).
- **Validation:** Type-safe runtime input parsing schemas using `zod`.
- **Interactive Documentation:** Autogenerated Swagger UI docs hosted natively at `/api/v1/docs`.

### Frontend (Supportive Dashboard UI)
- **Framework:** Vite + React + TypeScript.
- **Styling:** Premium Custom **Vanilla CSS** design system. Uses HSL variables, deep dark gradients, glassmorphism (`backdrop-filter`), smooth hover animations, and card transformations. *No third-party utility packages or UI frameworks were used.*
- **API Interaction:** Axios client with automated JWT injection interceptors and global 401 session expiration handling.
- **Role-Based Views:**
  - **Standard Users:** Can create, edit, filter, search, cycle status, and delete their own tasks.
  - **Admin Users:** Can inspect a system-wide user directory, toggle account access levels (`USER` vs `ADMIN`), and view/delete tasks across all accounts.

## Quick Start & Setup

Ensure you have **Node.js (v18+)** and **npm** installed.

### 1. Install Dependencies
At the root of the project, run:
```bash
npm install && npm install --prefix backend && npm install --prefix frontend
```

### 2. Configure Environment Variables
A default `.env` has been created for you at `backend/.env`:
```env
PORT=5000
JWT_SECRET=super-secret-token-key-change-in-production-12345!
DATABASE_URL="file:./dev.db"
NODE_ENV=development
```

### 3. Initialize the Database & Run Migrations
Run the Prisma migrations from the `backend/` folder to generate the local SQLite database and client types:
```bash
cd backend
npx prisma migrate dev --name init
cd ..
```

### 4. Run the Applications
Run both backend API and React frontend concurrently with a single command from the root folder:
```bash
npm run dev
```
- **React Frontend:** http://localhost:5173
- **Express Backend:** http://localhost:5000
- **Swagger Documentation:** http://localhost:5000/api/v1/docs

### 5. Programmatic API Integration Test
To run the automated verification script verifying JWT tokens, route guards, validation, and role limits, run:
```bash
cd backend
node verify_api.js
```

## Scalability Note

To scale this application to support millions of active users, the following architectural additions should be implemented:

### 1. Database Scaling (Replacing SQLite with Postgres)
- **Database Selection:** Switch to a high-concurrency relational database (e.g., **PostgreSQL** or **Amazon Aurora**).
  - Update `schema.prisma` connection provider from `sqlite` to `postgresql` and update the connection URL.
- **Read/Write Splitting (Replication):** Direct all write transactions (task creation/updates) to a Primary master node and distribute read queries (fetching task lists) across multiple Read Replicas using connection poolers like **PgBouncer** to prevent socket starvation.
- **Sharding & Partitioning:** Partition the `Task` table by `userId` (hash sharding). This guarantees that queries for a single user's task list execute locally on a single database partition rather than performing cross-node table scans.

### 2. Caching Layer (Redis)
- **Response Cache:** Store frequently queried, semi-static data (like user profiles and active tasks list) in **Redis**.
  - Cache-aside Strategy: When a user fetches their tasks, check Redis. If there's a cache miss, fetch from PostgreSQL and write to Redis with a TTL (Time To Live, e.g., 10 minutes).
  - Eviction: Invalidate the user's task cache key on write actions (`POST`, `PUT`, `DELETE`).
- **Session Caching:** Offload JWT blocklists or token revocations to a high-speed Redis key-value cluster.

### 3. Load Balancing & Reverse Proxies
- **Ingress Controller:** Deploy **Nginx** or **HAProxy** as a reverse proxy/load balancer.
- **Load Balancing Algorithms:** Route client requests using Round Robin or Least Connections to multiple stateless instances of the Express service API running inside private subnets.
- **SSL Termination:** Offload HTTPS decryption to the Load Balancer (e.g., AWS ALB) so that backend Express processes are dedicated solely to business logic and validation.

### 4. Microservices Decomposition
As features grow, decompose this modular monolith into independent microservices:
1. **Identity & Auth Service:** Dedicated to password hashing, user registration, role audits, and JWT signing.
2. **Tasks & Operations Service:** Scaled independently to handle high-frequency CRUD transactions.
- **API Gateway:** Implement an API Gateway (e.g., **Kong** or **Express Gateway**) to act as a single entryway. The Gateway handles rate-limiting, CORS, global authentication verification, and routes requests to the respective backend microservice.

### 5. Containerization & Deployment
- **Dockerization:** Create lightweight Docker containers for both backend and frontend.
- **Orchestration:** Deploy onto **Kubernetes (EKS/GKE)**. Configure Horizontal Pod Autoscalers (HPA) to scale backend node instances based on CPU utilization and request thresholds.
- **CDN (Content Delivery Network):** Deploy the static build assets of the Vite React client directly to **AWS S3** and distribute globally using **CloudFront** (CDN) to ensure sub-millisecond load times.
